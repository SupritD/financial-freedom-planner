1. DDD Architecture Overview
The Financial Freedom Planner follows a strict 4-layer Domain-Driven Design architecture. Business logic lives exclusively in the Domain layer. The Presentation layer is thin — controllers validate via Form Requests and delegate to Application Actions immediately.

1.1 Four-Layer Architecture
Layer
Location
Responsibility
Presentation Layer
app/Http/Controllers
app/Http/Resources
routes/
Receives HTTP requests. Delegates immediately to Application layer. Zero business logic. Returns API Resources or Blade views.
Application Layer
src/Domain/{Name}/Actions
src/Domain/{Name}/Services
src/Domain/{Name}/DTOs
Orchestrates use cases. Calls Domain services and repositories. Handles transactions. Dispatches Events. No SQL or HTTP knowledge.
Domain Layer
src/Domain/{Name}/Models
src/Domain/{Name}/Contracts
src/Domain/{Name}/ValueObjects
src/Domain/{Name}/Events
Pure business logic. Domain Models, Value Objects, Domain Events, Repository Contracts (interfaces). Zero framework dependency.
Infrastructure Layer
src/Domain/{Name}/Repositories
app/Jobs
app/Listeners
app/Notifications
Implements Domain Contracts. Eloquent repositories, queue jobs, external APIs, file storage. Framework-dependent but domain-decoupled.

1.2 DDD Principles Applied
Principle
Implementation in this Project
Ubiquitous Language
Every class, method, variable uses domain terminology. "GoalContribution" not "GoalPayment". "EmergencyFundDeposit" not "FundAdd".
Bounded Contexts
12 domains are isolated. Income domain knows nothing about Investment domain. Cross-domain communication via Events or Application Services only.
Dependency Inversion
Domain layer defines Repository interfaces (Contracts). Infrastructure layer implements them. Domain never imports Eloquent directly.
Value Objects
Money amounts wrapped in Money value object. Immutable. BCMath arithmetic inside. Prevents raw float usage across the codebase.
Domain Events
State changes dispatch Domain Events (GoalMilestoneReached, BudgetExceeded). Listeners handle side effects (notifications, cache, audit).
Aggregate Roots
FinancialGoal is an aggregate root — owns GoalContributions. SavingAccount owns SavingTransactions. Debt owns AmortizationSchedule.
Repository Pattern
All data access through Repository interfaces. Application layer is testable without DB. Infrastructure swappable (e.g., from Eloquent to Doctrine).
CQRS (Soft)
Read models (financial_snapshots, expense_trends) are separate from write models. Dashboard reads from snapshots, not live aggregate queries.

1.3 Request Lifecycle
FLOW
HTTP Request  ->  Route  ->  Middleware (Auth, Tenant, RateLimit, Idempotency)
  ->  FormRequest (validation)  ->  Controller (thin — 5 lines max)
  ->  Action (use-case orchestrator)  ->  Service (domain logic)
  ->  Repository (data access)  ->  Model (Eloquent)
  ->  Domain Event dispatched  ->  Listeners (notifications, cache, audit)
  ->  API Resource (response formatting)  ->  HTTP Response

3. Domain Folder Template
Every bounded domain follows this identical folder structure. Consistency across domains means any developer can navigate any domain immediately. The structure enforces DDD layer separation at the filesystem level.

src/Domain/{DomainName}/
|
|-- Actions/                          # Use-case orchestrators (1 class = 1 use case)
|   |-- Create{Entity}Action.php      # e.g. CreateIncomeEntryAction.php
|   |-- Update{Entity}Action.php
|   |-- Delete{Entity}Action.php
|   `-- {SpecificUseCase}Action.php   # e.g. RecordSipTransactionAction.php
|
|-- Contracts/                        # Repository interfaces (no Eloquent here)
|   |-- {Entity}RepositoryInterface.php
|   `-- {Entity}ReadRepositoryInterface.php   # Separate read model contract
|
|-- DTOs/                             # Data Transfer Objects (typed input/output)
|   |-- {Entity}Data.php              # Input DTO from FormRequest
|   `-- {Entity}SummaryData.php       # Output DTO for API Resources
|
|-- Events/                           # Domain events
|   |-- {Entity}Created.php
|   |-- {Entity}Updated.php
|   `-- {DomainEvent}Occurred.php    # e.g. BudgetExceeded.php
|
|-- Exceptions/                       # Domain-specific exceptions
|   |-- {Entity}NotFoundException.php
|   `-- {DomainRule}ViolationException.php   # e.g. InsufficientFundsException.php
|
|-- Models/                           # Eloquent models (domain models)
|   |-- {Entity}.php                  # Primary entity model
|   `-- {RelatedEntity}.php
|
|-- Repositories/                     # Eloquent implementations of Contracts
|   |-- Eloquent{Entity}Repository.php
|   `-- Eloquent{Entity}ReadRepository.php
|
|-- Services/                         # Domain services (complex business logic)
|   |-- {DomainName}Service.php       # Primary domain service
|   `-- {Calculation}Service.php      # e.g. GoalProjectionService.php
|
|-- ValueObjects/                     # Immutable domain value types
|   |-- Money.php                     # BCMath-backed monetary value
|   |-- Percentage.php
|   `-- DateRange.php
|
`-- Enums/                            # PHP 8.1+ backed enums
    |-- {Entity}StatusEnum.php        # e.g. GoalStatusEnum.php
    `-- {CategoryType}Enum.php

3.1 Folder Responsibility Map
File / Class
Responsibility
Actions/
Single-responsibility use-case classes. Called by Controllers. Orchestrate Services and Repositories. Wrap in DB transactions where needed.
Contracts/
PHP interfaces for repositories. Domain layer depends on these — never on Eloquent directly. Enables swapping storage implementations.
DTOs/
Typed data transfer objects. FormRequests create Input DTOs. Actions return Output DTOs. No raw arrays passed between layers.
Events/
Immutable domain events dispatched when state changes. Carry the minimum data needed by listeners. Named in past tense: BudgetExceeded.
Exceptions/
Business rule violations as typed exceptions. Caught by exception handler and converted to HTTP responses. DomainException base class.
Models/
Eloquent models with no business logic inside them. Only relationships, casts, scopes, and fillable. Logic lives in Services.
Repositories/
Eloquent implementations of Contract interfaces. All SQL queries live here. No SQL anywhere else in the application.
Services/
Complex domain calculations and business rules. Stateless. Injected into Actions. e.g. GoalProjectionService, XirrCalculationService.
ValueObjects/
Immutable typed wrappers. Money wraps BCMath. Prevents primitive obsession. Two Money instances are equal if same amount + currency.
Enums/
PHP 8.1+ backed enums for status fields, type classifications. Replaces magic string constants across the codebase.

6.1 Middleware Responsibilities
File / Class
Responsibility
ResolveTenant.php
Reads subdomain from Host header. Looks up tenant in cache then DB. Binds Tenant to request. Returns 404 if tenant not found or inactive.
IdempotencyMiddleware.php
Reads Idempotency-Key header. Checks idempotency_keys table. If found and valid: return cached response. If missing: return 400 on mutation endpoints.
RateLimitByTier.php
Applies different rate limits per route group: Auth=5/min, Financial writes=30/min, Reports=3/min, Bulk=10/hr. Uses Redis for counters.
SecurityHeaders.php
Adds: X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy, Permissions-Policy. Applied to all responses.
NoCacheFinancialData.php
Adds Cache-Control: no-store, no-cache, must-revalidate to all financial data API responses. Prevents browser and CDN caching.
RequireConsent.php
Checks consent_records for data_processing consent before allowing financial data routes. Returns 403 with consent_required message if withdrawn.
AdminIpAllowlist.php
Reads ADMIN_ALLOWED_IPS from config. Returns 403 if request IP not in allowlist. Applied to all /api/v1/admin/* routes.
StructuredRequestLogger.php
Logs every request as structured JSON: {request_id, user_id, tenant_id, method, path, status, duration_ms}. Masks PII fields (email, phone, amounts).

6.2 Middleware Stack Order
ORDER
Global (all requests): StructuredRequestLogger -> SecurityHeaders -> NoCacheFinancialData
API routes: ResolveTenant -> EnsureTenantIsActive -> Authenticate -> EnsureEmailIsVerified
Financial mutation routes: + IdempotencyMiddleware -> RateLimitByTier -> RequireConsent
Admin routes: + RequireAdminRole -> AdminIpAllowlist
Auth routes only: + RateLimitByTier (5/min) [no Authenticate middleware]

11. Models Registry
Every Eloquent model lives in its domain's Models/ folder. Models contain: relationships, casts, scopes, fillable/guarded, and trait declarations ONLY. Zero business logic inside models — all logic in Services and Actions.

All money columns cast to Money value object via custom Eloquent cast.
All models with financial data have: HasLedgerEntry trait (auto-creates ledger entries on save).
All user-data models have: BelongsToTenant trait (auto-injects tenant_id scope on all queries).

16. Naming Conventions
Consistent naming is a DDD requirement — it enables ubiquitous language across code, docs, and conversations. Every team member should be able to guess a class name from its description and vice versa.

Class Type
Convention + Examples
Actions
Verb + Noun + Action: CreateIncomeEntryAction, RecordDebtPaymentAction, GenerateMonthlyReportAction
Services
Noun + Service: GoalProjectionService, AmortizationService, XirrCalculationService
Repositories
Eloquent + Entity + Repository: EloquentGoalRepository, EloquentDebtPaymentRepository
Contracts
Entity + RepositoryInterface: GoalRepositoryInterface, DebtRepositoryInterface
DTOs
Entity + Data (input) or EntitySummary + Data (output): GoalData, GoalSummaryData
Events
Past tense noun phrase: GoalCompleted, BudgetExceeded, DebtPaymentRecorded
Listeners
Verb + Object + Listener: SendGoalCompletedEmailListener, WriteLedgerEntryListener
Jobs
Verb + Object + Job: ComputeXirrForInvestmentJob, GenerateMonthlyReportJob
Notifications
Noun + Notification: GoalMilestoneNotification, EmiReminderNotification
Controllers
Entity + Controller: IncomeEntryController, AmortizationScheduleController
Requests
Http verb + Entity + Request: StoreExpenseRequest, UpdateGoalRequest
Resources
Entity + Resource/Collection: InvestmentResource, InvestmentCollection
Policies
Entity + Policy: DebtPolicy, FinancialGoalPolicy
Commands
Verb + Object + Command: ComputeHealthScoresCommand, SendEmiRemindersCommand
Exceptions
Noun + Exception: GoalNotFoundException, InsufficientSavingBalanceException
Enums
Entity + StatusEnum or TypeEnum: GoalStatusEnum, AssetClassEnum
ValueObjects
Domain noun: Money, Percentage, FinancialYear, DeviceFingerprint, TenantId
Traits
Has + Feature: HasUlid, HasAuditLog, HasLedgerEntry, BelongsToTenant

RULE
No generic names: No "Manager", "Handler", "Helper", "Util" classes. Every name describes its exact responsibility.
No abbreviations: "GoalContribution" not "GoalContrib". "AmortizationSchedule" not "AmorSched".
Domain language first: use terms from the financial domain, not technical terms. "recordPayment" not "savePayment".
One class per file, one responsibility per class. File name === class name (PSR-4).
